Tool Mentor: Tool Support
This guide shows how to use the tools that are proposed in this methodology.
Main Description

The Architecture Model, the Variability Model, the Requirements Model, and the set of Tactic Models are specified in the XML Metadata Interchange (XMI) format. They are processed by the software infrastructure provided in the Eclipse Modeling Framework (EMF) to specify and execute queries against them at runtime. The Variability Model and the set of Tactic Models conform to the MOSKitt4SPL metamodel; the Architecture Model depends on the problem domain. In the case study it conforms to the SoaML metamodel, which extends the UML metamodel to support service modeling; and the Requirements Model conforms to the OpenOME metamodel. The Variability Model and the Architecture Model are taken as input to create the Weaving Model in the ATLAS Model Weaver Tool.

SALMon was chosen as the Context Monitor because its components are mostly technology-independent and they act as services, making its architecture customizable for our purpose. Context Conditions are specified as SPARQL Protocol and RDF Query Language (SPARQL) queries to the ontology in the Context Model. By using SPARQL queries, we have implemented the operations to insert new triples in the RDF graph of the ontology and to evaluate the values in the context model in order to find out if a predefined context condition has been accomplished. When an unknown context event arises, the Evolution Planner uses the generic rule reasoner in Jena to carry out forward chaining. In addition, the Evolution Planner uses the EMF Compare APIs to merge the Tactic Model into the Tactic Model. It also uses the EMF Model Query (EMFMQ) to activate or deactivate features in the evolved Variability Model during execution. The Execution Engine is implemented with Swordfish. Web services are created using the Java API for XML Web services (JAX-WS) and deployed as bundles in Swordfish.
