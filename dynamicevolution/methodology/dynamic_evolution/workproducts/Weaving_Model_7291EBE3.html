Description
Main Description

The features in the Variability Model are dynamically activated or deactivated in order to reach a particular configuration of the service composition. Therefore, it is necessary to count on a connection between the highly-abstract elements in the Variability Model and low-level system operations. Since the Architecture Model represents the operations in the system architecture at any moment, the definition of a bridge between the elements in the Variability Model and the elements in the Architecture Model could be used to support dynamic evolutions in the underlying system. As a result, if there is any change in the Variability Model, the mappings or links in this bridge could be queried at runtime in order to determine which changes need to be carried out in the Architecture Model.

In order to define this bridge, this methodology proposes the creation of a weaving model. A weaving model captures relationships between model elements, conforms to a weaving metamodel, and is not self contained (i.e., it is useful only if the related models also exist) [1]. The weaving model contains a set of links. Each link in the weaving model has two endpoints. The first endpoint refers to elements in the Variability Model (e.g. features) and in the set of Tactic Models. The second endpoint refers to elements in the Architecture Model.
