Tool Mentor: Tools
Main Description
ATLAS Model Weaver: The weaving models can be created with this tool. AMW supports only one source model and only one destination model. Therefore, at the implementation level we propose to map a unique XMI file that encapsulates the set of tactic models, expressed as feature models, to a unique XMI file that encapsulates the tactic models, expressed as composition models. URL: http://www.eclipse.org/gmt/amw/.

Babel: In the case of tactics that are based on composite Web services, Babel can be used to generate WS-BPEL code from tactic models expressed as BPMN models. Babel is an open-source tool that is the result of integrating a set of techniques to translate models captured using a core subset of BPMN into WS-BPEL (Ouyang et al., 2009). The proposed techniques are capable of generating readable WS-BPEL code by discovering “patterns” in the BPMN models that can be mapped onto WS-BPEL block-structured constructs or acyclic graphs of control links. The latest version of Babel can be downloaded as an Eclipse plugin. URL: http://www.bpm.scitech.qut.edu.au/research/projects/oldprojects/babel/tools/.

Eclipse Modeling Framework (EMF): The set of tactic models, which are expressed as composition models, can be created with the infrastructure provided by EMF. This step maps to the Create the Tactic Models (Expressed as Composition Models) activity in the process model to design dynamic evolutions. URL: http://www.eclipse.org/modeling/emf.
FMF-Based Configuration Tool: This tool can be used to complete the WS-BPEL code generated by Babel. FMF is a framework whose main goal is to facilitate the development of form-based editors that manipulate models. FMF is an extension of FEFEM. FMF is a part of the MOSKitt framework, in which the PROS has been constantly involved. A master thesis describes the FMF-based configuration tool (De La Fuente, 2012). Also, there is a video demonstration: https://vimeo.com/58989214.

MOSKitt4SPL: The tactic models, which are expressed as feature models, can be created with our MOSKitt4SPL tool. This step maps to the Create the Tactic Models (Expressed as Feature Models) activity in the process model to design dynamic evolutions. URL: http://www.pros.upv.es/m4spl.

Requirements Model: OpenOME can be used to create the requirements model because it offers an open metamodel that can be used to create XMI-based requirements models. OpenOME is an open source, general, goal-oriented and/or agent-oriented modeling and analysis tool. It provides users with a graphical interface to develop models, and supports access to a knowledge base that allows for sophisticated computer-aided analysis. OpenOME supports the i* framework.

In our approach, we manage four kinds of OpenOME contributions: 1) the MakeContribution is a positive contribution strong enough to satisfice a softgoal (i.e., ++); 2) the HelpContribution is a partial positive contribution, not sufficient by itself to satisfy the softgoal (i.e., +); 3) the BreakContribution is a negative contribution sufficient enough to deny a softgoal (i.e., - -); and 4) the HurtContribution is a partial negative contribution, not sufficient by itself to deny the softgoal (i.e., -). Although the OpenOME metamodel contains the Belief metaclass, its graphical interface misses the representation for beliefs or claims. Therefore, it is necessary to use the EMF generic editor in order to add beliefs or claims into the requirements model.

STP BPMN Modeler: The set of tactic models, which are expressed as composition models, can be created with the STP BPMN Modeler. This step maps to the Create the Tactic Models (Expressed as Composition Models) activity in the process model to design dynamic evolutions. URL: http://git.eclipse.org/c/bpmnmodeler.

Text Editor: A text editor can be used to implement the fragments of WS-BPEL code that invoke service operations implementing tactics. We propose to store each invocation instruction in a separated text file. Each text file also contains the information about the partner link of the tactic's service, which will participate in the service composition. The pieces of code for invoking the tactic's services will eventually be merged into the running service composition to face unknown context events in the open world. Other approaches can also be followed to implement the fragments. For example, it is possible to store all invocation instructions in a database or to count on a single text file with all the invocation instructions. The key aspect in this task is to count on a fast mechanism to retrieve these instructions at runtime.

A text editor can be also used to implement the rules file.

Web Services Project: We propose the use this project to implement atomic Web services. The Web Services Project has two components. First, the JST Web services component contains tools for developing and interacting with Java Web services. Second, the WST Web services component contains tools for Web services development which is not Java specific. A demonstration of the Web Services Project is available online. In this demonstration, a top-down approach is followed to create an atomic Web service (i.e., the Web service is created from a WSDL file). URL: http://www.eclipse.org/webtools/ws.
